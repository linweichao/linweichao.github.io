工作队列（Work Queue）

（使用Java Client）

在第一个教程中我们写了从一个命名的队列中发送和接收消息的程序。在这篇中我们会创建*工作队列（Work Queue）*用来在多个工作者（worker）之间分发耗时的任务。

工作队列（亦称任务队列（Task Queue））背后的思想是避免立即做一个资源密集的任务，并且需要等待它完成。取而代之的是我们安排这个任务稍后再做。我们封装*任务（task）*作为一个消息然后发送到队列。一个运行在后台的工作者进程会取出（pop）任务并最终执行这个工作。当你运行多个工作者，任务会在它们之间共享。

这个概念在web程序中特别有用，不可能在一个短暂的HTTP请求窗口中处理一个复杂的任务。

准备

在这个教程的上一部分我们发送了一个包含”Hello World!“的消息。现在我们会发送代表复杂任务的字符串。我们没有真实的任务，像调整图片大小或渲染pdf文件，因此让我们假装很忙 - 通过使用Thread.sleep()功能。我们会把字符串中的点数量作为它的复杂性；每个点会解释为一秒钟的”工作“。例如，由Hello…描述的伪造任务会花3秒钟。

我们会从前面例子中的Send.java代码中些微地修改下，允许任意消息从命令行中发送。这个程序会调度任务到工作队列，因此让我们把它命名为NewTask.java：

    for (char ch: task.toCharArray()) {
        if (ch == '.') Thread.sleep(1000);
    }

循环（round-robin）分发

使用任务队列的一个优点是简单并行任务的能力。如果我们积压了一些工作，我们只需要添加更多的工作者，这种方式，简单扩展。

首先，让我们尝试同时运行两个工作者实例。它们都会从队列获取消息，但准确来说是怎么样的？让我们看一行。

默认，RabbitMQ会发送消息到下一个消费者，依次。平均每个消费者会得到相同数量的消息。这种分发消息的方式叫做循环（round-robin）。试一下三个或更多工作者。

消息确认

做一个任务要花几秒钟。你可能想知道如果消费者之一开始一个长时间的任务，只是部分的完成然后死亡时会发生什么。用我们现在的代码，一旦RabbitMQ分发消息给消费者，它立即把它标记为删除。在这种情况，如果你杀掉一个工作者我们会丢失它刚刚处理的消息。同样也会丢失所有分发到这个工作者但是还没处理的消息。

但是我们不想丢失任何任务。如果一个工作者死亡了，我们希望任务可以分发到其它工作者。

为了保证消息决不会丢失，RabbitMQ支持消息确认。一个ack（确认）由消费者发送回来告诉RabbitMQ一个特定的消息已经被接收，处理，RabbitMQ可以自由删除它。

如果一个消费者死亡（channel关闭，connection关闭或TCP连接丢失）而没有发送ack，RabbitMQ会推断一个消息没有完全处理，会重新如队。如果同时有其它消费者在线，它会快速重新分发给其它消费者。这种方式你可以确认没有消息丢失，即使工作者偶然间死亡。

没有任何的消息超时；RabbitMQ会在消费者死亡时重新分发消息。即使处理一个消息花费很长，很长时间也没事。

手动消息确认默认打开。之前的例子我们通过autoAck=true标记明确关闭了。是时候把这个标记设置成false然后从工作者中发送一个合适的确认，一旦完成了任务。

channel.basicQos(1); // accept only one unack-ed message at a time (see below)

DeliverCallback deliverCallback = (consumerTag, delivery) -> {
  String message = new String(delivery.getBody(), "UTF-8");

  System.out.println(" [x] Received '" + message + "'");
  try {
    doWork(message);
  } finally {
    System.out.println(" [x] Done");
    channel.basicAck(delivery.getEnvelope().getDeliveryTag(), false);
  }
};
boolean autoAck = false;
channel.basicConsume(TASK_QUEUE_NAME, autoAck, deliverCallback, consumerTag -> { });

使用这段代码我们可以确认即使使用CTRL+C在工作者处理消息时杀掉它，不会丢失任何东西。工作者死亡后所有未确认的消息很快会重新分发。

确认必须在接收分发相同的channel中发送。尝试使用不同channel确认导致channel级别的协议异常。查看确认的文档指南学习更多。

忘记确认

忘记basicAck是一个很常见的错误。它是一个简单的错误，但后果是严重的。当你的客户端退出时消息会重新分发（看起来像随机分发），但是RabbitMQ会占用更多内存，由于它不能释放任何未确认的消息。

为了调试这种类型的错误你可以使用rabbitmqctl来打印messages_unacknowledged字段：

1

sudo rabbitmqctl list_queues name messages_ready messages_unacknowledged

或者在Windows中，去掉sudo：

1

rabbitmqctl.bat list_queues name messages_ready messages_unacknowledged

消息持久性

我们已经学习了如何确保即使消费者死亡，任务也不会丢失。但是如果RabbitMQ服务器停止我们的任务仍然会丢失。

当RabbitMQ退出或崩溃它会忘记队列和消息，除非你告诉它不要。确保消息不丢失需要两件事情：需要标记队列和消息为持久的。

首先，我们需要确保队列在一个RabbitMQ节点重启后存活。为了达到这个目的，我们需要声明它为持久的（durable）：

1
2

boolean durable = true;
channel.queueDeclare("hello", durable, false, false, null);

尽管这个命令自身是正确的，依我们现在的配置它不会生效。这个因为我们已经定义了一个叫做hello的非持久性队列。RabbitMQ不允许使用不同参数重新定义已存在的队列，对试图这么做的程序将会返回错误。但是有一个快速的变通方法 - 让我们定义一个不同名字的队列，例如task_queue：

1
2

boolean durable = true;
channel.queueDeclare("task_queue", durable, false, false, null);

queueDeclare的改变需要应用到消费者和生产者的代码。

这时候，我们确定task_queue队列即使RabbitMQ重启也不会丢失。现在我们需要标记我们的消息为持久化的 - 通过设置MessageProperties(实现了BasicProperties)值为PERSISTENT_TEXT_PLAIN。

1
2
3
4
5

import com.rabbitmq.client.MessageProperties;

channel.basicPublish("", "task_queue",
            MessageProperties.PERSISTENT_TEXT_PLAIN,
            message.getBytes());

消息持久性的说明

标记消息为持久的不能完全保证消息不会丢失。尽管它告诉RabbitMQ把消息保存到硬盘，当RabbitMQ接收消息和没有保存时仍然有一个短暂的时间窗口。同样，RabbitMQ没有为每个消息做fsync(2) – 可能只是保存到缓存没有真正写入硬盘。持久性的保证是不健壮的，但是它对于我们简单的任务队列足够了。如果你需要更强的保证那么你可以使用publisher confirms。

公平分发

你可能注意到分发仍然没有精确按我们所想的来工作。例如，一个有两个工作者的情况，当所有奇数消息是沉重的，偶数消息是轻量的，一个工作者会持续忙碌，另一个几乎不做任何工作。嗯，RabbitMQ不知道关于这些的任何事，仍然均匀地分发消息。

这会发生是因为RabbitMQ只是当消息进入队列时分发消息。它不会查看一个消费者未确认消息的数量。它只是盲目地分发第n个消息给第n个消费者。

为了克服这个我们可以使用basicQos方法，带有prefetchCount = 1 的设置。这告诉RabbitMQ一次不要给一个工作者超过一个消息。或者，换句话说，不要分发新的消息给工作者，直到它处理和确认了前面的一个。取而代之，它会分发这个消息到下一个不忙碌的工作者。

1
2

int prefetchCount = 1;
channel.basicQos(prefetchCount);

关于队列大小的注意事项

如果所有的工作者都很忙，你的队列会被装满。你会想要留意这，可能添加更多工作者，或者有其它策略。

import com.rabbitmq.client.Channel;
import com.rabbitmq.client.Connection;
import com.rabbitmq.client.ConnectionFactory;
import com.rabbitmq.client.MessageProperties;

public class NewTask {

  private static final String TASK_QUEUE_NAME = "task_queue";

  public static void main(String[] argv) throws Exception {
    ConnectionFactory factory = new ConnectionFactory();
    factory.setHost("localhost");
    try (Connection connection = factory.newConnection();
         Channel channel = connection.createChannel()) {
        channel.queueDeclare(TASK_QUEUE_NAME, true, false, false, null);

        String message = String.join(" ", argv);

        channel.basicPublish("", TASK_QUEUE_NAME,
                MessageProperties.PERSISTENT_TEXT_PLAIN,
                message.getBytes("UTF-8"));
        System.out.println(" [x] Sent '" + message + "'");
    }
  }

}

import com.rabbitmq.client.Channel;
import com.rabbitmq.client.Connection;
import com.rabbitmq.client.ConnectionFactory;
import com.rabbitmq.client.DeliverCallback;

public class Worker {

  private static final String TASK_QUEUE_NAME = "task_queue";

  public static void main(String[] argv) throws Exception {
    ConnectionFactory factory = new ConnectionFactory();
    factory.setHost("localhost");
    final Connection connection = factory.newConnection();
    final Channel channel = connection.createChannel();

    channel.queueDeclare(TASK_QUEUE_NAME, true, false, false, null);
    System.out.println(" [*] Waiting for messages. To exit press CTRL+C");

    channel.basicQos(1);

    DeliverCallback deliverCallback = (consumerTag, delivery) -> {
        String message = new String(delivery.getBody(), "UTF-8");

        System.out.println(" [x] Received '" + message + "'");
        try {
            doWork(message);
        } finally {
            System.out.println(" [x] Done");
            channel.basicAck(delivery.getEnvelope().getDeliveryTag(), false);
        }
    };
    channel.basicConsume(TASK_QUEUE_NAME, false, deliverCallback, consumerTag -> { });
  }

  private static void doWork(String task) {
    for (char ch : task.toCharArray()) {
        if (ch == '.') {
            try {
                Thread.sleep(1000);
            } catch (InterruptedException _ignored) {
                Thread.currentThread().interrupt();
            }
        }
    }
  }
}

(Worker.java的源码)

使用消息确认和prefetchCount你可以设置一个工作队列。持久性的选项使任务即使RabbitMQ重启也会存活。
